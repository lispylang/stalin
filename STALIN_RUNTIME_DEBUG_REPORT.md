# Stalin Runtime Debug Report
**Date:** October 1, 2025
**Platform:** macOS Darwin 25.0.0 / ARM64 (Apple Silicon)
**Stalin Binary:** stalin-native (ARM64, 3.0 MB)

---

## Executive Summary

**ROOT CAUSE IDENTIFIED:** Stalin runtime crashes with segmentation fault (EXC_BAD_ACCESS) during initialization in function `f1120`. The crash occurs before any Scheme code is parsed, indicating a fundamental runtime initialization bug on ARM64 architecture.

---

## Debugging Process

### Issue #1: PATH Problem (✅ SOLVED)

**Symptom:**
```
sh: stalin-architecture-name: command not found
Argument to STRUCTURE-REF is not a structure of the correct type
```

**Root Cause:**
Stalin binary tries to execute `stalin-architecture-name` script but cannot find it because `./include` is not in PATH.

**Solution:**
```bash
export PATH="$(pwd)/include:$PATH"
./stalin-native -On -I ./include -c program.sc
```

**Status:** ✅ FIXED - Stalin now runs with proper PATH

---

### Issue #2: Segmentation Fault (⚠️ BLOCKER)

**Symptom:**
```
Error
```

**Investigation:**
```bash
export PATH="$(pwd)/include:$PATH"
./stalin-native -On -I ./include -c hello-simple.sc
Error  # Generic error with no details
```

**Root Cause (via lldb):**
```
Process stopped
* thread #1, queue = 'com.apple.main-thread'
  stop reason = EXC_BAD_ACCESS (code=1, address=0x6fdfbb80)

frame #0: 0x000000010007f08c stalin-native`f1120 + 980
stalin-native`f1120:
->  0x10007f08c <+980>: ldr    x8, [x8]      # ← CRASH HERE
    0x10007f090 <+984>: ldr    w9, [x8, #0x8]
    0x10007f094 <+988>: cmp    w9, #0x438
    0x10007f098 <+992>: b.ne   0x10007f5e4
```

**Analysis:**
- Crash occurs in `f1120` function at offset +980
- Attempting to dereference pointer at invalid address 0x6fdfbb80
- This appears to be an uninitialized or corrupted pointer
- Crash happens during runtime initialization, before parsing Scheme code

**Likely Causes:**
1. **Architecture Mismatch:** ARM64 is being treated as IA32 by stalin-architecture-name
2. **Memory Layout:** Data structure sizes differ between IA32 and ARM64
3. **Pointer Alignment:** ARM64 has stricter alignment requirements
4. **Structure Tags:** The tag comparison (cmp w9, #0x438) fails because structure layout is wrong

## Technical Details

### Architecture Detection Issue

**Current behavior:**
```bash
$ ./include/stalin-architecture-name
IA32  # ← WRONG! Should be ARM64 or AMD64
```

**Code in stalin-architecture-name:**
```sh
aarch64/*|arm64/*)
    echo "IA32";;  # Workaround: Stalin doesn't have ARM64 def
```

**Problem:** Stalin was compiled with IA32 assumptions but runs on ARM64 hardware.

**Impact:**
- Structure sizes mismatch (32-bit vs 64-bit)
- Pointer sizes wrong (4 bytes vs 8 bytes)
- Alignment requirements different
- Tag values incorrect

### Memory Access Pattern

The crash at `f1120+980` shows:
```assembly
ldr x8, [x8]  # Load 64-bit value from address in x8
```

This suggests Stalin is trying to follow a linked structure (list, tree, etc.) but the pointer is:
- Not initialized (garbage value)
- Pointing to unmapped memory
- Corrupted by structure layout mismatch

### Function f1120

Without source code, we can infer `f1120` is likely:
- Runtime initialization function
- Structure type checking/validation
- Symbol table setup
- Type system bootstrap

The tag comparison `cmp w9, #0x438` (1080 decimal) suggests checking for a specific structure type tag.

## Workaround: Use Pre-Generated C Files

Since Scheme→C compilation is broken, we use pre-generated C files:

```bash
# Given: program.c (generated by Stalin on another system)
./cosmocc/bin/cosmocc -o program program.c \
  -I./include -L./include -lm -lgc -lstalin \
  -O3 -fomit-frame-pointer -fno-strict-aliasing

./program  # ✅ WORKS - Runs as universal binary
```

**Status:** ✅ WORKING PERFECTLY
